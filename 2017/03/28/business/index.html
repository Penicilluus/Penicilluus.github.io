微服务代码结构（Golang）

微服务设计原则

  • 单一责任原则（Single Responsibility Principle，SRP）
  • 确保证微服务设计能支持服务的敏捷/独立地开发和部署。

微服务之间的通信

  • 同步通信，restful、thrift、grpc
  • 异步通信，rabbitmq、kafka
  • 消息格式，json、protobuf
  • 服务协议，swagger、IDL

微服务的架构模式

  • 点对点风格-直接调用服务
  • API Gateway风格
  • 消息代理风格，pub/sub

数据管理

  • 去中心化的数据管理

服务注册与服务发现

  • 客户端发现，或者服务端发现（k8s）

微服务部署

  • 能够横向扩展，无状态部署
  • 一个服务崩溃不能影响其他服务崩溃
  • docker将应用打包成镜像，部署为容器，使用k8s管理docker

安全

  • 使用oauth2的授权模式
  • 独立出oauth服务

如何使用配置或者环境变量，发现服务
服务依赖的问题，分清楚基础业务与复合业务
微服务的目录结构
client 提供给第三方的直接函数调用，封装底层grpc service调用，读取配置连接服务器，
service 下分dao、server、manager，dao层主要跟数据库交互，server是grpc server接口的实现，manager是中间层 server –> manager –> dao
serviceImpl { mysqlConnection } serviceImpl { manager } manager { mysqlConnection redisConnection}
subscriber 主要是kafka消费服务，引入消息队列主要是为了异步通讯，解耦业务服务。所有的消息都是单向的，都假定是一定是成功的，不成功则有相应的解决方案
使用with function结构，将要调用的函数当做参数传递，将重复逻辑复用。例如grpc客户端的调用，可统一埋点。

protocol（grpc pbfile gen-go）、foundation(公共代码)、openapi（第三方接口）
服务发现，服务解耦，服务层级划分，如果需要强一致性，可以将该服务的范围放大，例如金钱操作

账户系统

  • 注册（手机号、验证码、密码）验证验证码、验证密码、创建用户、创建access_token
  • 登录（手机号、密码）
  • 第三方登录（微信）（客户端调用微信登录界面，微信回调后台接口并携带code，后台接收code，换取access_token）
  • 刷新access_token 返回新的access_token和refresh_token， refresh_token比access_token的有效期更长，当refresh_token过期，重新登录

微信登录流程

  1. 用户点击微信登录，调用第三方接口，进入授权页面，与后台无交互
  2. 用户点击授权，回调后台，将code参数传入后台
  3. 后台根据code、AppID和AppSecret 换取access_token
  4. access_token的有效期是2小时，refresh_token的有效期是30天
  5. 通过access_token获取用户头像、昵称等信息

access_token 策略 (使用redis保存token)

  1. user-access_token（用户对应access_token），key 是用户，保存的是token
  2. access_token user- access_token对应用户，key是token，保存的是struct实体
  3. access_token refresh策略，当token过期，使用refresh_token刷新（返回新的access_token,refresh_token）
    注：当access_token过期，并不会影响用户操作行为，自动判断，并调用refresh_token的接口，返回新的access_token和refresh_token(微信是否会返回新的refresh_token？)当refresh_token过期，则说明30天内用户都无任何行为，则直接让用户重新登录。

支付系统

一般支付流程

  1. 用户点击微信支付，应用发送pay请求到后台，sign签名
  2. 应用收到后台的sign签名，调起第三方支付页面
  3. 用户与第三方支付交互，支付失败或成功返回应用
  4. 调用query_order查询支付结果，后台与第三方交互
  5. 第三方回调notify，支付成功或失败
